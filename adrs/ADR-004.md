# ADR-004: Rillan Adopts a Plugin Architecture for Community Extensibility

## Status

Proposed

## Context

Rillan is evolving from a personal proxy daemon into an open-source routing kernel for LLM interaction. The goal is to let the community extend Rillan in ways its authors haven't anticipated -- new LLM backends, prompt transformations, skills, retrieval strategies, middleware, and integrations with tools like opencode. Think "oh-my-zsh for LLM routing": a small, solid kernel with a thriving plugin ecosystem.

Today, Rillan has clean internal interfaces (Provider, Classifier, Evaluator, Recorder, QueryEmbedder, QueryRewriter) but every implementation is compiled in. Markdown skills are cataloged but have no execution model. MCP endpoints are registered but never called. Project config fields like `system_prompt` and `instructions` are parsed but unused. The seams exist; the extension surface does not.

The tension: extensibility invites complexity, and complexity kills small projects. Rillan needs a plugin model that is simple enough that a first-time contributor can ship a plugin in an afternoon, powerful enough that it doesn't become the bottleneck for ambitious integrations, and safe enough that a plugin can't silently exfiltrate credentials or bypass policy.

## Decision

Rillan introduces a plugin system built on four extension surfaces, a manifest contract, a discovery convention, and a security boundary.

### Extension surfaces

Each surface corresponds to a lifecycle phase of a request or operation. A plugin registers for one or more surfaces.

#### 1. Providers (`provider` surface)

A provider plugin adds an LLM backend. It receives chat completion requests and returns responses.

**Contract:** stdin/stdout JSON-RPC over stdio, or an HTTP endpoint that accepts OpenAI-compatible `/v1/chat/completions` requests and returns OpenAI-compatible responses.

**Registration:**

```yaml
# ~/.config/rillan/plugins/anthropic-native.yaml
kind: provider
id: anthropic-native
transport: stdio
command: ["rillan-provider-anthropic"]
capabilities: ["chat", "reasoning", "tool_calling"]
```

Or for HTTP-based providers (anything already running an OpenAI-compatible server):

```yaml
kind: provider
id: vllm-local
transport: http
endpoint: http://localhost:8000/v1
capabilities: ["chat"]
```

The `rillan llm add` command gains a `--plugin` flag that references an installed plugin instead of a built-in type. The named provider registry stays the same; the backend is just delegated to the plugin process.

**Why this matters:** This is the single most important surface. It lets the community add Anthropic, Gemini, Mistral, Cohere, local GGUF inference, or anything else without waiting for upstream merges. The HTTP transport means any existing OpenAI-compatible server is already a valid provider plugin with zero code.

#### 2. Middleware (`middleware` surface)

A middleware plugin intercepts requests and responses at the HTTP boundary. It can inspect, transform, or reject traffic.

**Contract:** each middleware is a command that reads a JSON envelope on stdin and writes a (possibly modified) envelope to stdout. The envelope includes the request body, headers, and Rillan-internal metadata (project classification, policy trace, matched provider). A middleware can:

- Modify the request before it reaches the provider (prompt rewriting, context injection, PII scrubbing)
- Modify the response before it reaches the client (output filtering, formatting)
- Reject the request by returning an error envelope

**Registration:**

```yaml
kind: middleware
id: pii-scrubber
phase: pre_request    # pre_request | post_response | both
command: ["rillan-middleware-pii", "--mode", "redact"]
priority: 100         # lower runs first
```

**Chain execution:** middleware runs in priority order. If any middleware returns an error, the chain halts and the error propagates to the client. This is the same model as HTTP middleware stacks (Express, Gin, etc.) but over stdio so plugins can be written in any language.

**Why this matters:** this is where "oh-my-rillan" lives. Community members can ship prompt enhancers, cost trackers, compliance filters, logging integrations, A/B testers, and caching layers as standalone commands.

#### 3. Skills (`skill` surface)

A skill plugin adds a new capability the agent can invoke. This extends the current markdown skill catalog with executable skills.

**Contract:** a skill is either:

- **Markdown (passive):** a prompt template injected into agent context when the skill is active. This is what skills are today -- the existing catalog already handles install, checksum, and metadata. The change is that markdown skills are actually wired into the chat completions handler via `system_prompt` and `instructions` injection.

- **Executable (active):** a command that receives a structured JSON invocation on stdin and returns structured JSON results on stdout. The agent decides when to invoke it based on the skill's declared capabilities.

**Registration (executable):**

```yaml
kind: skill
id: web-search
transport: stdio
command: ["rillan-skill-web-search"]
capabilities:
  - "search the web for current information"
triggers:
  - "when the user asks about recent events"
  - "when the query requires information newer than the training cutoff"
inputs:
  - name: query
    type: string
    required: true
outputs:
  - name: results
    type: array
```

**Why this matters:** executable skills turn Rillan from a passthrough proxy into an agent runtime. The agent can call out to web search, database queries, file operations, code execution, or anything else a plugin author ships.

#### 4. Hooks (`hook` surface)

A hook plugin reacts to lifecycle events without modifying the request/response flow. Hooks are fire-and-forget (async) by default, or synchronous when gating is needed.

**Events:**

| Event | When | Sync option |
|-------|------|-------------|
| `request.received` | HTTP request arrives | no |
| `request.routed` | Provider selected | no |
| `request.completed` | Response sent to client | no |
| `skill.invoked` | Agent invokes a skill | yes (can gate) |
| `proposal.created` | Agent proposes an action | yes (can approve/deny) |
| `config.changed` | Runtime config mutated | no |
| `index.completed` | Index rebuild finishes | no |

**Registration:**

```yaml
kind: hook
id: slack-notifier
events: ["proposal.created", "request.completed"]
command: ["rillan-hook-slack", "--channel", "#llm-ops"]
sync: false
```

**Why this matters:** hooks are the integration glue. Slack notifications, Datadog metrics, audit forwarding, cost tracking dashboards, and CI/CD triggers all fit naturally as async hooks. Synchronous hooks on `proposal.created` let external systems participate in the approval gate.

### Plugin manifest and discovery

Every plugin is a directory containing a `plugin.yaml` manifest:

```yaml
name: rillan-plugin-anthropic
version: 0.1.0
description: "Native Anthropic provider for Rillan"
author: "community-contributor"
license: "MIT"
min_rillan_version: "0.9.0"

surfaces:
  - kind: provider
    id: anthropic-native
    transport: stdio
    command: ["rillan-provider-anthropic"]
    capabilities: ["chat", "reasoning"]
```

**Discovery order:**

1. `~/.config/rillan/plugins/*/plugin.yaml` -- user-installed plugins
2. `<project>/.rillan/plugins/*/plugin.yaml` -- project-local plugins
3. Built-in plugins compiled into the Rillan binary (existing provider, skills)

`rillan plugin install <path-or-url>` copies a plugin directory into the user plugin path. `rillan plugin list`, `rillan plugin remove`, `rillan plugin enable/disable` manage the registry.

### Security boundary

Plugins run as child processes, not loaded into the Rillan process. This gives us:

- **Process isolation:** a plugin crash doesn't take down the daemon
- **Language independence:** plugins can be Go, Python, Rust, shell scripts, or anything with stdin/stdout
- **Credential boundary:** plugins never see keyring contents directly. If a plugin needs an API key, the user stores it via `rillan plugin login <id>` (same keyring path as providers), and Rillan passes it as an environment variable to the child process. Plugins cannot enumerate or access other credentials.
- **Policy enforcement:** middleware plugins receive the policy trace but cannot override it. A plugin cannot escalate a `VerdictBlock` to `VerdictAllow`. Middleware can only tighten restrictions, not loosen them.
- **Timeout and resource limits:** plugin invocations have configurable timeouts (default 30s for skills, 5s for middleware, 60s for providers). Hung plugins are killed.

**What plugins cannot do:**

- Access the keyring directly
- Bypass policy verdicts
- Modify the plugin registry (self-replication)
- Run without being explicitly installed and enabled
- Exceed their declared surface (a skill plugin cannot intercept HTTP middleware)

### Configuration integration

Plugins participate in the existing config hierarchy:

- **Global:** `rillan plugin enable/disable` controls which plugins are active daemon-wide
- **Project:** `.rillan/project.yaml` can restrict which plugins are allowed per-repo (same pattern as `llm_allowed` and `mcp_enabled`)
- **System policy:** system config can blocklist plugin IDs or require signature verification

```yaml
# .rillan/project.yaml
plugins:
  allowed: ["anthropic-native", "pii-scrubber"]
  denied: ["*"]  # explicit allowlist mode
```

### Relation to MCP

MCP endpoints registered via `rillan mcp add` are treated as a special case of the skill surface. When the MCP client is implemented, each registered MCP server's tools become available as agent-invocable skills with the same security boundary (process isolation, timeout, read-only default). The `rillan mcp` CLI remains the registration path; the plugin system provides the execution model.

## Consequences

- Rillan becomes a routing kernel rather than a monolithic proxy. The core binary stays small; capabilities grow through the ecosystem.
- The stdio JSON contract means plugins can be written in any language. A Python developer can ship a provider plugin without knowing Go.
- Process isolation means plugin bugs don't crash the daemon, and the security boundary is enforceable without trusting plugin code.
- The four-surface model (provider, middleware, skill, hook) covers the extension points we know about today while being narrow enough that each surface has a clear, testable contract.
- Community plugins like "oh-my-rillan" theme packs become possible: a curated set of middleware, skills, and hooks that users install with one command.
- The `system_prompt`, `instructions`, and `routing` config fields that exist today gain a reason to exist -- they configure how the kernel composes plugin behavior.
- The cost is a new subprocess management layer, a plugin registry, and contract versioning. This is real complexity, but it's bounded complexity with clear failure modes (process exit, timeout, malformed JSON), unlike in-process plugin architectures where failure modes are unbounded.
- Plugin contract versions must be maintained carefully. Breaking changes to the stdin/stdout envelope require a major version bump and a migration path.
- Testing plugins requires a harness that simulates Rillan's side of the stdio contract. Rillan should ship `rillan plugin test <path>` to validate a plugin against its declared surface before installation.
