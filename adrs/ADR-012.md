# ADR-012: Tokenizer-Backed Outbound Code Minimization

## Status

Proposed

## Context

Rillan already has bounded retrieval, policy enforcement, audit tracing, and local-model-assisted query rewriting. Remote-bound requests can be constrained by retrieval settings such as `top_k` and `max_context_chars`, and policy can already force local-only routing or redact sensitive material before provider dispatch.

Those controls are useful, but they still treat outbound code size mostly as a chunk-count or character-budget problem. That is not precise enough for model-facing request costs, and it does not distinguish between code that is operationally necessary for the question and code that is merely verbose. Comments are a common source of this inefficiency: some comments are critical because they carry API contracts, generated-code markers, kubebuilder/php-style annotations, build directives, or security caveats; others are irrelevant to the current question and consume outbound budget without improving the answer.

Rillan needs a structural minimization layer that runs after retrieval compilation but before remote provider dispatch. That layer must remain local-first, deterministic, reversible enough for debugging and reintegration, and compatible with the existing policy, audit, and retrieval seams. It also needs to respect the repository's current priorities:

- OpenAI-compatible upstreams are the first exact provider path.
- Local Ollama may be used for local reasoning helpers, but should degrade gracefully when unavailable.
- Audit should record hashes and trace metadata rather than raw request bodies.
- Repo and system policy must remain able to tighten behavior without silent bypass.

## Decision

Rillan will introduce a tokenizer-backed outbound code minimization path with the following rules.

1. **Token budgeting happens before remote egress.**
   - Remote-bound code payloads are measured with a real tokenizer instead of relying only on character limits.
   - The first exact budgeting target is the OpenAI-compatible provider family.
   - Budgeting is pinned by tokenizer encoding, not only by model name, so token accounting stays deterministic even if model aliases drift.

2. **Code minimization is deterministic, not generated.**
   - The minimization layer operates on retrieved code context after retrieval compilation and before outbound provider dispatch.
   - The transformation logic is deterministic and source-aware.
   - Local or remote LLMs must not directly rewrite source text in the minimization path when reversibility matters.

3. **Comment handling is explicit and user-configurable.**
   - Outbound code minimization will support explicit comment modes:
     - `include_all`
     - `include_relevant`
     - `include_annotations`
     - `exclude_all`
   - License notices, generated-code markers, build directives, and policy-required comments are preserved regardless of weaker heuristics when the runtime contract requires them.
   - Annotation-only behavior covers machine-meaningful comment forms such as php-style `@` annotations, kubebuilder markers, pragma/build comments, and similar allowlisted patterns.

4. **Local Ollama is advisory only for comment relevance.**
   - Ollama may classify whether a comment is relevant to the current user question.
   - Ollama is used only for `include_relevant` mode.
   - Classification results are bounded and confidence-gated.
   - If Ollama is unavailable, times out, or returns low-confidence output, the system falls back to deterministic rules and configured defaults rather than failing the request.

5. **Minimized code must remain reconstructable.**
   - The minimization layer records source-span mappings from original content to minimized output.
   - Reconstruction metadata is sufficient to explain or rehydrate comment-preserving original code when needed for debugging, reintegration, or audit correlation.
   - For Go-specific source rewrites, a comment-preserving transformation approach may be used where necessary; otherwise line/byte-span mapping remains the default safe contract.

6. **Short-lived translation artifacts may be stored locally.**
   - Minimized translation artifacts may be stored in local SQLite-backed state for short-term reuse.
   - Any vector-assisted reuse of minimized artifacts is optional and must never bypass exact compatibility checks on source hash, settings, and span identity.
   - These artifacts are operational cache data, not a second durable source of truth for repository source.

7. **Audit and policy stay authoritative.**
   - Audit records hashes, source references, minimization mode, token counts, and artifact identifiers rather than raw minimized code bodies by default.
   - Policy evaluation remains able to constrain or block remote egress regardless of minimization settings.
   - Project-level configuration may tune minimization behavior, but must not loosen machine-level policy guarantees.

## Consequences

- Rillan gains a model-facing budget that is more faithful than character counts and can make outbound minimization decisions before remote dispatch.
- Comment handling becomes an explicit product behavior instead of an accidental side effect of truncation.
- Local Ollama contributes useful semantic judgment without becoming a structural source transformer or a hard dependency for request success.
- Reconstruction and source-span mapping add implementation complexity, but they keep the minimization path explainable and safer to debug.
- Exact token accounting will be strongest for OpenAI-compatible providers first; other provider families may need conservative fallback budgets until they gain dedicated tokenizer support.
- Short-lived minimized artifact reuse can reduce repeated work for nearby requests, but it introduces invalidation and retention concerns that must be handled deliberately.
- Config, docs, and audit surfaces will need updates so users can understand what was minimized and why.
