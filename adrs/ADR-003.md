# ADR-003: Milestone-Two Indexing Uses One Explicit Root and Embedded SQLite Storage

## Status

Accepted

## Context

Milestone two introduces the first local knowledge substrate for Rillan. The goal is to make indexing deterministic, local-first, and easy to rebuild without adding background watchers, external vector services, or retrieval-time complexity too early.

## Decision

Rillan uses these milestone-two indexing defaults:

- index one explicit local root at a time
- trigger indexing manually through `rillan index`
- expose current state through `rillan status`
- resolve a relative `index.root` against the config file location so one config owns one deterministic local root
- store metadata, chunks, and vectors in a single embedded SQLite database under the local data directory
- keep text and code artifacts in scope first; skip binary content
- use deterministic chunk IDs and a full-rebuild indexing strategy in the first cut
- keep vector storage behind a narrow internal seam, but store vectors in the same SQLite database for now

Filesystem ownership for this milestone:

- the config file owns runtime settings and root selection only
- the local data directory owns `index/index.db`, including schema version records, index run history, document metadata, chunk bodies, and vector records
- the log directory stays separate and is not part of the indexed corpus by default

## Consequences

- Milestone two stays operationally simple because it does not introduce a second local daemon or watcher lifecycle.
- Index rebuilds are deterministic and easy to reason about because the first cut replaces the full corpus on each run.
- Retrieval can build on a stable local schema in milestone three without changing the filesystem model.
- The vector abstraction remains upgradeable later if approximate search or a richer SQLite vector extension becomes necessary.
