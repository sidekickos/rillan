# ADR-002: Milestone-One Runtime Defaults Stay Local and User-Scoped

## Status

Accepted

## Context

Milestone one is a local-first daemon for developer machines on macOS and Linux. It needs predictable defaults for API binding and filesystem layout without pulling in service-management or remote-access complexity too early.

## Decision

Rillan uses these milestone-one runtime defaults:

- bind the daemon to `127.0.0.1:8420`
- write starter config through `rillan init`
- keep config, data, and logs as separate local concerns
- use user-local config and data directories by default
- treat the daemon as the intended model egress path for product usage, without claiming host-level enforcement yet

Default locations:

- macOS config: `~/Library/Application Support/rillan/config.yaml`
- Linux config: `${XDG_CONFIG_HOME:-~/.config}/rillan/config.yaml`
- macOS data: `~/Library/Application Support/rillan/data`
- Linux data: `${XDG_DATA_HOME:-~/.local/share}/rillan`
- macOS logs: `~/Library/Logs/rillan`
- Linux logs: `${XDG_STATE_HOME:-~/.local/state}/rillan/logs`

## Consequences

- The default developer experience stays local and predictable.
- `rillan serve` is safe by default because it does not bind to all interfaces.
- Future service packaging for `launchd` and `systemd` can override paths without changing local-development semantics.
- Later vector and local-model runtime assets have a clear place to live without polluting config files.
